## 压缩后带宽（NVENC 在线编码）

### 编码参数选取

```
目标质量：ML 可用（PSNR ≥ 40dB），保留像素细节
H.264 目标码率：15 Mbps / 路（1080p@60fps），720p@30fps 按比例降低
H.265 目标码率：8 Mbps / 路（等效质量，约 H.264 一半）
深度流目标码率：3-5 Mbps / 路（低分辨率、低 fps）
```

## 磁盘写入速度需求

| 场景 | 写入速度 | 是否为瓶颈 |
|------|---------|---------|
| H.264 压缩后写入 | ~23 MB/s | **不是**（普通 SATA SSD 500+ MB/s） |
| H.265 压缩后写入 | ~13 MB/s | **不是** |
| 原始帧写入（不建议） | ~4,352 MB/s | **是**，需 RAID-0 NVMe |

**结论：使用 NVENC 在线压缩，磁盘带宽不是瓶颈。**

## 真正的瓶颈：接口带宽与 GPU 编码

### 接口带宽规划

| 接口 | 带宽 | 承载设备 | 原始流量 |
|------|------|---------|---------|
| 10GbE NIC | 1.25 GB/s | 工业相机 ×8（片上 JPEG 后） | ~288 MB/s |
| USB3 Gen2 ×1 | 1.25 GB/s | D405 #1（彩色 + 深度） | ~100 MB/s |
| USB3 Gen2 ×1 | 1.25 GB/s | D405 #2（彩色 + 深度） | ~100 MB/s |
| USB3 Gen2 ×1 | 1.25 GB/s | D435（彩色 + 深度） | ~230 MB/s |
| USB3 Gen2 ×1 | 1.25 GB/s | Fisheye #1 | ~534 MB/s |
| USB3 Gen2 ×1 | 1.25 GB/s | Fisheye #2 | ~534 MB/s |

> Fisheye 1080p@90fps 原始流量 534 MB/s 接近 USB3 Gen2 上限（1.25 GB/s），
> 若相机支持片上压缩（MJPEG）可大幅降低接口压力。

### GPU 编码能力

| GPU | NVENC session 限制 | 是否满足 16 路并发 |
|-----|-------------------|-----------------|
| RTX 3080/3090 | 3 路（驱动软限制） | 否，需 2 块 |
| RTX 4090 | 无限制（40 系移除限制） | **是，单卡覆盖** |
| RTX A4000/A5000（专业卡） | 无限制 | 是 |

## 单机硬件最低配置建议

```
CPU:  16 核以上（pipeline 线程调度、parquet 写入）
GPU:  RTX 4090 × 1
      （16 路 NVENC 并发编码，去掉 session 限制是关键）
RAM:  64 GB
      （相机 frame buffer 约 1 GB，其余用于系统和 Python 运行时）
NIC:  10GbE × 1（工业相机 GigE 汇聚）
USB:  PCIe USB3 Gen2 扩展卡 × 2（提供 8 个独立 Gen2 口）
SSD:  NVMe 4TB（主存储）+ NVMe 1TB（系统盘）
主板:  需要 4 个 PCIe 插槽（GPU + NIC + USB卡 × 2）
       建议 X299 / TRX50 / HEDT 工作站平台
```

## LeRobot v3 格式适配说明

### dataset fps 选择

推荐 `recording.fps = 90`（Fisheye 原生频率）：

| fps 选择 | Fisheye | RGB/工业相机 | 存储开销 |
|---------|---------|------------|---------|
| 60Hz | 丢 33% 帧 | 原生 | 最小 |
| **90Hz（推荐）** | **原生无损** | **33% 帧重复** | 中（+50%） |
| 180Hz（LCM） | 2× 重复 | 3× 重复 | 最大（×2-3） |

RGB 重复帧用 `rgb_is_duplicate` 列标记（parquet 自定义列），训练时可过滤。

### 触觉数据存储方式

Tactile 1000×3 uint8（3KB/帧）在 LeRobot v3 中的存储选项：

| 方式 | 优点 | 缺点 |
|------|------|------|
| parquet array 列 | 原生支持，读写简单 | 列宽固定，压缩率一般 |
| 单独 parquet 文件（sidecar） | 可独立读取，不影响主 dataset | 需自定义 dataloader |
| 小分辨率视频（32×32 灰度） | 压缩率高，与图像流一致 | 需要转换，不直观 |

**推荐**：parquet array 列，与 `encoder` 和 `pose` 列共存于同一 parquet 文件。

### 配置示例

```json
{
  "recording": {
    "fps": 90,
    "format": "lerobot_v3",
    "use_videos": true,
    "video_codec": "h264"
  }
}
```

`recording.py:96-97` 读到显式 `fps` 后直接返回，无需修改代码。

## H.264 vs H.265 选择

**结论：用 H.264。**

### 决策依据

录制时写盘是一次性操作，23 MB/s 和 13 MB/s 对 NVMe 都无所谓。
训练时每个 epoch 要把所有视频帧解码一遍，解码速度才是瓶颈。

| 维度 | H.264 | H.265 |
|------|-------|-------|
| 单集存储 | ~1.4 GB | ~0.8 GB |
| NVENC 编码负载 | 低 | 高 2-3x |
| CPU 解码速度 | 快 | 慢 2-3x |
| NVDEC 硬件解码 | 两者接近 | 两者接近 |
| 工具兼容性 | 全部支持 | 部分有专利问题 |
| LeRobot / torchvision / decord | 完整支持 | 完整支持 |

### 本系统的具体情况

训练时需同时解码 13 路视频流，DataLoader 默认走 CPU 解码：

```
13 路 × 60fps = 780 帧/秒 解码吞吐需求
H.265 CPU 解码约为 H.264 的 2x 计算量
→ DataLoader worker 容易成为训练瓶颈
```

4TB NVMe 存 H.264 约 2,800 集，存储不是紧张因素。

### 例外场景

仅在以下情况考虑 H.265：
- 存储容量严重受限（< 1TB 且需存 5,000 集以上）
- 训练侧能确保全程使用 NVDEC 硬件解码（需在 DataLoader 中显式启用，LeRobot 默认不走硬件解码）
